## 1. CI/CD Pipeline Standard

Mọi pipeline CI/CD **bắt buộc** đi qua đủ các bước sau theo thứ tự:

```
Code → Build → Test → Scan → Deploy
```

### Các bước scan bắt buộc (phải pass trước khi deploy):
| Bước | Công cụ / Tool | Mục đích |
|------|---------------|----------|
| **SAST scan** | SonarQube, Snyk, v.v. | Quét mã nguồn tĩnh tìm lỗi bảo mật |
| **Dependency scan** | Snyk, OWASP Dependency-Check | Kiểm tra thư viện bên thứ 3 tìm lỗ hổng |
| **Docker artifact scan** | Trivy, Snyk Container | Quét container image trước khi push / deploy |

## 2. Observability Standard

Hệ thống **bắt buộc** thiết lập observability đầy đủ dựa trên 5 trụ cột:

| Trụ cột | Yêu cầu |
|---------|---------|
| **Logging** | `Structured logging` dạng JSON; log đủ context (`request_id` / `trace_id`, user ID, service name, operation); **không log** password, token, PII, secret |
| **Metrics** | Thu thập chỉ số định lượng về sức khoẻ hệ thống (CPU, memory, request rate, error rate, latency) |
| **Tracing** | End-to-end distributed tracing — xác định được lỗi qua các service |
| **Uptime** | Theo dõi thời gian hoạt động của từng service |
| **Alerting** | Cài đặt alert dựa trên **SLO** đã được định nghĩa và đồng thuận với đội sở hữu service |

### 2.1 Log Format Template (JSON — bắt buộc)

Mọi log line phải tuân theo format JSON sau. Không thêm field bổ sung tùy tiện — mọi extension phải có ADR.

```json
{
  "timestamp": "2026-04-18T10:00:00.123Z",
  "level": "INFO",
  "service": "order_service",
  "environment": "production",
  "trace_id": "abc123",
  "request_id": "req-uuid",
  "user_id": "user-uuid",
  "operation": "createOrder",
  "message": "Order created successfully",
  "duration_ms": 142,
  "http_status": 201,
  "error_code": null,
  "error_message": null
}
```

**Quy tắc bắt buộc:**
- `timestamp`: ISO 8601 UTC, có milliseconds
- `service`: tên service (snake_case, không có version number)
- `trace_id`: OpenTelemetry trace ID — truyền xuyên suốt qua tất cả services
- `request_id`: ID của HTTP request / message cụ thể — dùng để correlate logs trong cùng 1 request
- `user_id`: masked nếu là context background job; `null` nếu không có authenticated user
- Khi error: `error_code` là mã lỗi từ Error Code Catalog của api-specs; `error_message` là message hiển thị (không phải stack trace)
- Stack trace: log riêng ở `ERROR` level, KHÔNG lẫn vào structured field — dùng `"stack_trace": "..."` field tách biệt

### 2.2 Log Level Semantics

- `ERROR`: lỗi chặn luồng hoặc gây ảnh hưởng tới người dùng / nghiệp vụ, cần action rõ ràng từ đội sở hữu hoặc on-call.
- `WARN`: bất thường chưa chặn luồng; chỉ alert hoặc page theo threshold, tần suất, hoặc khi vi phạm SLO.
- `INFO`: mốc trạng thái và tracing; không tự tạo alert.

### 2.3 Correlation ID Header Convention

HTTP header names (case-insensitive per RFC 7230, nhưng dùng nhất quán):

| Header | Nguồn | Mục đích |
|--------|-------|---------|
| `X-Request-ID` | Generated by API Gateway hoặc client | Unique ID per HTTP request — không thay đổi qua retries |
| `X-Trace-ID` | Generated by OpenTelemetry / tracing SDK | Distributed trace ID — span across services |
| `X-Correlation-ID` | Forward từ upstream caller | Business-level correlation (VD: job batch ID, saga ID) |

**Rules:**
- API Gateway tạo `X-Request-ID` nếu client không gửi
- `X-Trace-ID` được tạo và inject bởi OpenTelemetry SDK — không hardcode
- Mọi downstream HTTP call phải forward `X-Request-ID` và `X-Trace-ID` trong outbound headers
- Mọi async event/message phải include `trace_id` và `request_id` trong event envelope (xem `events-template.md`)

### 2.4 Metric Naming Convention

Format: `{service_name}_{domain}_{action}_{unit}` (Prometheus / OpenMetrics style, snake_case, lowercase)

| Metric type | Pattern | Ví dụ |
|-------------|---------|-------|
| HTTP request duration | `{service}_http_request_duration_seconds` | `order_service_http_request_duration_seconds` |
| HTTP request total | `{service}_http_requests_total` | `order_service_http_requests_total` |
| HTTP error rate | `{service}_http_errors_total` | `order_service_http_errors_total` |
| DB query duration | `{service}_db_query_duration_seconds` | `order_service_db_query_duration_seconds` |
| DB connection pool | `{service}_db_pool_{metric}` | `order_service_db_pool_active_connections` |
| Cache hit/miss | `{service}_cache_hits_total` / `{service}_cache_misses_total` | `order_service_cache_hits_total` |
| Queue depth | `{service}_{queue_name}_queue_depth` | `order_service_payment_queue_depth` |
| Background job duration | `{service}_{job_name}_job_duration_seconds` | `order_service_invoice_gen_job_duration_seconds` |

**Rules:**
- Labels: `{method}`, `{status_code}`, `{route}` for HTTP; `{operation}` for DB; `{cache_name}` for cache
- Histograms for latency: use standard buckets `[0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]` seconds
- Counters for totals: always suffix `_total`
- Gauges for current state: `_active`, `_pending`, `_depth`

### 2.5 Alerting Threshold Examples (default baselines — adjust per NFR targets)

| Alert | Condition | Severity | Action |
|-------|-----------|----------|--------|
| High error rate | HTTP 5xx rate > 1% over 5 min | CRITICAL | Page on-call |
| Elevated error rate | HTTP 5xx rate > 0.1% over 15 min | WARNING | Slack notify |
| High latency | p95 latency > NFR target × 1.5 over 5 min | WARNING | Slack notify |
| Critical latency | p95 latency > NFR target × 2 over 2 min | CRITICAL | Page on-call |
| DB pool exhaustion | Active connections > 80% of pool max | WARNING | Slack notify |
| High queue depth | Queue depth > expected 10-min throughput | WARNING | Slack notify |
| Service down | Health check fails 3 consecutive times | CRITICAL | Page on-call |
| DLQ spike | DLQ message count > 10 in 5 min | WARNING | Slack notify + ticket |

> **Quan trọng**: Thresholds trên là baseline — `nfr-v{X}.md §8` là nguồn chân lý. Config alerting theo NFR targets của project, không dùng hardcoded defaults này.

### 2.6 Correlation IDs (legacy — giữ để backward compat)

- Mọi log line phải có ít nhất một correlation key: `request_id` hoặc `trace_id`; background jobs thêm `job_id` khi phù hợp.
- Correlation ID phải được truyền xuyên qua HTTP request, async message/event, background job, và webhook callback.
- Xem §2.3 cho header convention đầy đủ.

## 3. Environment Management

| Nguyên tắc | Mô tả |
|-----------|-------|
| **Infrastructure as Code** | Mọi tài nguyên hạ tầng phải được định nghĩa bằng code (Terraform, Ansible) — đảm bảo nhất quán, truy vết, tái sử dụng |
| **Immutable Deployment** | Không thay đổi cấu hình trực tiếp trên hệ thống đang chạy (`không hotfix trực tiếp trên production`) — thay thế bằng phiên bản mới hoàn toàn |
| **Configuration Management** | Mọi thay đổi config phải được theo dõi đầy đủ và có phê duyệt trước khi apply |
| **GitOps** | Git là "single source of truth" cho cả cấu hình và trạng thái deployment |

## 4. Secret Management

**Nơi lưu & phân loại secret tuân theo `security-standards.md § 2.1 Secret Management`** (nguồn chuẩn): runtime secrets → Hashicorp Vault / Google Secret Manager; CI/CD secrets ngắn hạn → secret store của CI platform; tuyệt đối không hardcode trong source code.

Phần vận hành riêng của DevSecOps:

- Mọi secret được nạp qua **environment variable** hoặc **secret manager API** — không qua file cấu hình tĩnh được commit.
- Chọn CI secret store khớp câu hỏi "đội dùng CI platform nào?" ở §1.

## 5. Cost Monitoring & Optimization

- **Right-sizing**: thiết kế hướng đến quy mô phù hợp với tải thực tế — tránh cấp phát thừa và cấp phát thiếu
- **Resource tagging**: mọi tài nguyên cloud phải được tag theo: cost center, business unit, product — để theo dõi chi phí phát sinh
- **Idle resource recovery**: có cơ chế theo dõi và thu hồi tài nguyên nhàn rỗi
- **Chargeback model**: khi xây dựng hệ thống dùng chung cho nhiều đơn vị, phải có mô hình tính giá thành dịch vụ và cơ chế phân bổ / charge ngược lại cho đơn vị sử dụng

## 7. Feature Flags / Progressive Rollout

Tooling: LaunchDarkly / Unleash / Statsig / inhouse — chọn theo budget + scale.

Bắt buộc:
- **Naming convention:** `<scope>.<feature>.<purpose>` ví dụ `checkout.new_payment.enable`.
- **Lifecycle:** development → staging → prod-canary (1–5%) → prod-rollout (25 / 50 / 100%) → archive (xóa flag + dead code) trong < 90 ngày.
- **Kill switch:** mọi feature có khả năng gây sự cố production phải có kill switch flag.
- **A/B testing:** flag tích hợp với analytics; experiment có hypothesis + success metric trước launch.
- **Audit:** flag config change ghi log + reviewer.
